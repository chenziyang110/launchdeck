# Launchdeck v0.1 Protocol

## Goal

The v0.1 protocol gives any local project a small, explicit lifecycle contract. Launchdeck does not need to know the stack if the project tells it which commands are authoritative.

## File

Launchdeck looks upward from the current directory for one of:

- `.launchdeck.yml`
- `.launchdeck.yaml`
- `launchdeck.yml`
- `launchdeck.yaml`

`.launchdeck.yml` is recommended for project-local lifecycle configuration.

## Schema

```yaml
version: 1
project:
  name: my-project
tasks:
  build:
    command: npm run build
    cwd: .
    env:
      NODE_ENV: production
    longRunning: false
    ports: []
    risk: low
  dev:
    command: npm run dev
    longRunning: true
    ports: [5173]
    risk: medium
clean:
  safe:
    - dist
    - node_modules/.vite
  risky:
    - node_modules
```

## Task Fields

- `command`: shell command to execute.
- `cwd`: task working directory relative to the project root. Defaults to `.`.
- `env`: additional environment variables.
- `longRunning`: whether Launchdeck should manage the process instead of waiting for it to exit.
- `ports`: expected local ports claimed by the task.
- `risk`: one of `low`, `medium`, `high`, or `destructive`.
- `description`: optional human-readable context.
- `log`: optional log file path relative to the project root for managed tasks.

Task values can also be short strings:

```yaml
tasks:
  build: npm run build
```

## Cleanup Fields

`clean.safe` contains paths that can be removed with `launchdeck clean --safe`.

`clean.risky` contains paths that require `launchdeck clean --all --yes`.

All clean paths must resolve inside the project root, and Launchdeck refuses to delete the project root itself.

## Runtime State

Managed task state is stored under:

```text
.launchdeck/
  logs/
  runtime/state.json
```

Each process record includes:

- task name
- command
- cwd
- pid
- ports
- log path
- started time
- status

## Agent Contract

Agents should prefer this sequence:

1. Run `launchdeck doctor --json`.
2. Use configured task names instead of inventing commands.
3. Start long-running tasks with `launchdeck dev` or `launchdeck start <task>`.
4. Inspect `launchdeck ps --json` before starting another server.
5. Read `launchdeck logs <task>` when diagnosing failures.
6. Stop managed processes with `launchdeck stop <task>`.
7. Use `launchdeck clean --safe` only for configured safe targets.
8. Require explicit user approval before `launchdeck clean --all --yes`.
